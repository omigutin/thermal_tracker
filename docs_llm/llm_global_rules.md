# LLM Global Rules

## Назначение документа

Этот документ описывает общие правила работы LLM с пользователем и Python-проектами.

Это не контекст конкретного проекта. Документ можно переиспользовать в разных репозиториях и периодически дополнять.

Перед выполнением проектной задачи LLM должна читать:

1. `docs_llm/llm_index.md`
2. `docs_llm/llm_global_rules.md`
3. `docs_llm/llm_project_context.md`
4. `docs_llm/llm_project_architecture.md`
5. `docs_llm/llm_project_handoff.md`
6. `docs_llm/llm_project_tasks.md`
7. `docs_llm/llm_worklog.md`, если файл существует

---

## Рабочий протокол по умолчанию

Перед изменением кода LLM обязана сначала подготовить:

1. анализ задачи;
2. первопричину проблемы, если это баг;
3. план решения;
4. список создаваемых и изменяемых файлов;
5. список создаваемых и изменяемых классов, методов, функций и ключевых сущностей;
6. оценку изменений публичного API;
7. риски;
8. проверки;
9. запрос подтверждения пользователя.

Без подтверждения пользователя код менять нельзя.

---

## Автономный режим

По умолчанию автономные изменения запрещены.

Исключение: пользователь или текущий `docs_llm/llm_project_tasks.md` может явно разрешить ограниченный автономный режим.

В автономном режиме LLM может выполнять только действия, прямо перечисленные в task-файле или в сообщении пользователя.

Если задача выходит за разрешённые границы, LLM обязана остановиться, зафиксировать проблему в отчёте и не делать спорное изменение.

Автономный режим не отменяет запреты на:

- `git push`;
- удаление файлов без явного разрешения;
- добавление зависимостей;
- изменение `poetry.lock`;
- изменение публичного API без фиксации и разрешения;
- массовые переименования без разрешения;
- изменение архитектуры вне текущего scope.

---

## Архитектура

LLM не должна делать заплатки вместо нормального решения.

Если проблема архитектурная, нужно:

1. объяснить первопричину;
2. предложить нормальное решение;
3. описать затронутые файлы;
4. описать риски;
5. дождаться решения пользователя, если текущая задача не разрешает автономное исправление этого класса проблем.

Временные решения и `TODO` по умолчанию запрещены.

Временный фикс можно предложить только как отдельный вариант, если баг критичный и блокирует запуск.

---

## Публичный API

API — Application Programming Interface / программный интерфейс приложения.

Если правильное решение требует изменения публичного API, нужно остановиться и отдельно описать:

- что меняется;
- почему старый API мешает;
- какие места проекта будут затронуты;
- варианты совместимости, адаптера, breaking change или legacy-слоя;
- риски.

Без согласования публичный API менять нельзя, кроме случаев, когда текущий task-файл явно разрешает конкретное изменение.

Даже если LLM исправляет явно битый публичный alias или export, это нужно отдельно отметить в отчёте как изменение публичной поверхности.

---

## Нейминг и структура

Нейминг считается частью архитектуры.

LLM не должна автоматически переименовывать публичные файлы, классы, методы, функции и сущности без согласования.

Если виден плохой нейминг, дублирование или перегруженная структура, нужно вынести это в отдельный блок замечаний и предложить варианты исправления.

Большие файлы и классы нежелательны. Если файл или класс разрастается, нужно предложить декомпозицию, но не выполнять её без подтверждения или явного разрешения в task-файле.

---

## Код

Обязательные правила:

- явная типизация аргументов, возвращаемых значений, атрибутов классов и важных переменных;
- русские docstring у классов, методов и функций;
- модульные docstring добавлять только если это требуется текущими правилами проекта;
- сообщения исключений на английском языке;
- комментарии только для объяснения причин решения, бизнес-логики, ограничений, производительности или компромиссов;
- не использовать `dict[str, Any]` как свалку, если структуру можно описать явно.

Использовать:

- `Enum` — для ограниченного набора вариантов;
- `dataclass` — для простых структур данных;
- обычный класс — если есть поведение и логика;
- `Protocol` — если нужен интерфейс без жёсткого наследования;
- `pydantic` — только если уже есть в проекте или согласован как зависимость.

---

## Валидация, ошибки и логирование

Валидировать данные нужно на границах системы:

- конфиги;
- CLI-параметры;
- API-запросы;
- файлы;
- пользовательский ввод;
- внешние данные от камеры, сервера или нейросети.

Если данные уже проверены и приведены к строгой модели, не нужно повторять одну и ту же проверку в каждом методе.

Не глушить исключения через пустой `except`.

Логировать только важное:

- запуск ключевых процессов;
- важные решения;
- ошибки;
- предупреждения;
- результаты крупных операций.

---

## Производительность

Если код связан с видео, ML, Computer Vision, сетью, сервером, файлами, большим количеством данных или realtime-обработкой, LLM должна учитывать:

- batching / пакетную обработку;
- очереди;
- асинхронность;
- потоки / процессы;
- изоляцию тяжёлых операций;
- лишнее копирование больших данных;
- узкие места.

При этом нельзя создавать избыточный сложный pipeline без причины.

---

## Конфигурации

Настройки должны быть отделены от бизнес-логики.

Архитектура должна позволять позже вынести настройки в файл, переменные окружения, базу данных, UI, CLI или API без переписывания ядра.

Не хардкодить проектные значения внутри бизнес-логики, если их можно вынести в конфигурацию.

---

## Тесты

Тесты пишутся на `pytest`.

LLM не должна автоматически писать тесты в процессе активного проектирования, если пользователь этого не просил.

Исключение: если текущий task-файл явно требует тесты, LLM должна добавить только тесты из scope задачи.

Структура тестов должна повторять структуру исходного кода.

Тесты должны проверять поведение, а не внутреннюю реализацию без причины.

---

## Документация

Документация по умолчанию пишется на русском языке.

Обычная документация проекта хранится в `docs/`.

Документация для LLM хранится отдельно в `docs_llm/`.

После каждого существенного этапа LLM должна обновить документы LLM, если текущая задача связана с автономной или многоэтапной работой.

---

## Зависимости

Не добавлять новые зависимости без согласования.

Если нужна новая библиотека, сначала написать:

```text
Новая зависимость:
- Название:
- Зачем нужна:
- Можно ли обойтись стандартной библиотекой:
- Есть ли уже похожая зависимость:
- Альтернативы:
- Рекомендация:
```

---

## Проверки качества

Запускать только те проверки качества, которые уже есть в проекте:

- `pytest`;
- `mypy`;
- `pylint`;
- `ruff`;
- `flake8`;
- `bandit`;
- настроенный форматтер.

Если инструмент не доступен в текущем окружении, не устанавливать его без согласования. Зафиксировать это в отчёте.

---

## Git

По умолчанию любые Git-команды считаются опасными и требуют разрешения.

Текущий task-файл может явно разрешить ограниченный набор локальных Git-команд.

Даже при разрешённых локальных коммитах запрещены:

```bash
git push
git pull
git reset
git rebase
git checkout
git clean
git stash
```

Перед коммитом LLM обязана проверить `git status` и не добавлять:

- `.env`;
- секреты, токены, ключи;
- `outputs/`;
- `sandbox/`;
- `__pycache__/`;
- `.pytest_cache/`;
- `.mypy_cache/`;
- `.ruff_cache/`;
- видео;
- веса моделей;
- датасеты;
- временные копии проекта.

Один этап = один локальный коммит, если коммиты разрешены текущей задачей.

---

## Итоговый отчёт

После изменений LLM должна написать отчёт:

```text
Итог:
- Что изменено:
- Почему так:
- Затронутые файлы:
- Проверки выполнены:
- Проверки не выполнены:
- Остаточные риски:
- Следующий рекомендуемый шаг:
```

Отчёт должен быть честным. Если проверка не запускалась, указать причину.
