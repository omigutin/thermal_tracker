# LLM Global Rules

## Назначение документа

Этот документ описывает общие правила работы LLM с пользователем и Python-проектами.

Это не контекст конкретного проекта. Документ можно переиспользовать в разных репозиториях и периодически дополнять.

Перед выполнением проектной задачи LLM должна читать:

1. `docs_llm/llm_global_rules.md`
2. `docs_llm/llm_project_context.md`
3. `docs_llm/llm_project_architecture.md`
4. `docs_llm/llm_project_handoff.md`
5. `docs_llm/llm_project_tasks.md`

---

## Рабочий протокол

Перед изменением кода LLM обязана сначала подготовить:

1. анализ задачи;
2. первопричину проблемы, если это баг;
3. план решения;
4. список создаваемых и изменяемых файлов;
5. список создаваемых и изменяемых классов, методов, функций и ключевых сущностей;
6. оценку изменений публичного API;
7. риски;
8. проверки;
9. запрос подтверждения пользователя.

Без подтверждения пользователя код менять нельзя.

---

## Архитектура и публичный API

LLM не должна делать заплатки вместо нормального решения.

Если проблема архитектурная, нужно объяснить первопричину, предложить нормальное решение, описать затронутые файлы, риски и дождаться решения пользователя.

Если правильное решение требует изменения публичного API, нужно остановиться и отдельно описать:

- что меняется;
- почему старый API мешает;
- какие места проекта будут затронуты;
- варианты совместимости, адаптера, breaking change или legacy-слоя;
- риски.

Без согласования публичный API менять нельзя.

---

## Нейминг и структура

Нейминг считается частью архитектуры.

LLM не должна автоматически переименовывать публичные файлы, классы, методы, функции и сущности без согласования.

Если виден плохой нейминг, дублирование или перегруженная структура, нужно вынести это в отдельный блок замечаний и предложить варианты исправления.

Большие файлы и классы нежелательны. Если файл или класс разрастается, нужно предложить декомпозицию, но не выполнять её без подтверждения.

---

## Код

Обязательные правила:

- явная типизация аргументов, возвращаемых значений, атрибутов классов и важных переменных;
- русские docstring у классов, методов и функций;
- модульные docstring добавлять только если это требуется текущими правилами проекта;
- сообщения исключений на английском языке;
- комментарии только для объяснения причин решения, бизнес-логики, ограничений, производительности или компромиссов;
- не использовать `dict[str, Any]` как свалку, если структуру можно описать явно.

Использовать:

- `Enum` — для ограниченного набора вариантов;
- `dataclass` — для простых структур данных;
- обычный класс — если есть поведение и логика;
- `Protocol` — если нужен интерфейс без жёсткого наследования;
- `pydantic` — только если уже есть в проекте или согласован как зависимость.

---

## Валидация, ошибки и логирование

Валидировать данные нужно на границах системы: конфиги, CLI/API-параметры, файлы, пользовательский ввод, внешние данные от камеры, сервера или нейросети.

Если данные уже проверены и приведены к строгой модели, не нужно повторять одну и ту же проверку в каждом методе.

Не глушить исключения через пустой `except`.

Логировать только важное: запуск ключевых процессов, важные решения, ошибки, предупреждения и результаты крупных операций.

---

## Производительность

Если код связан с видео, ML, Computer Vision, сетью, сервером, файлами, большим количеством данных или realtime-обработкой, LLM должна учитывать batching, очереди, асинхронность, потоки/процессы, изоляцию тяжёлых операций, лишнее копирование больших данных и узкие места.

При этом нельзя создавать избыточный сложный pipeline без причины.

---

## Конфигурации

Настройки должны быть отделены от бизнес-логики.

Архитектура должна позволять позже вынести настройки в файл, переменные окружения, базу данных, UI, CLI или API без переписывания ядра.

Не хардкодить проектные значения внутри бизнес-логики, если их можно вынести в конфигурацию.

---

## Тесты и документация

Тесты пишутся на `pytest`.

LLM не должна автоматически писать тесты в процессе активного проектирования, если пользователь этого не просил.

Структура тестов должна повторять структуру исходного кода.

Документация по умолчанию пишется на русском языке.

Обычная документация проекта хранится в `docs/`.

Документация для LLM хранится отдельно в `docs_llm/`.

---

## Зависимости, проверки и Git

Не добавлять новые зависимости без согласования.

Запускать только те проверки качества, которые уже есть в проекте.

Опасные действия требуют отдельного согласования:

- удаление файлов;
- массовое переименование;
- изменение `.env`, секретов, токенов и ключей;
- изменение миграций;
- изменение `poetry.lock`;
- изменение CI/CD;
- изменение публичного API;
- любые команды Git.

---

## Итоговый отчёт

После изменений LLM должна написать отчёт:

```text
Итог:
- Что изменено:
- Почему так:
- Затронутые файлы:
- Проверки выполнены:
- Проверки не выполнены:
- Остаточные риски:
- Следующий рекомендуемый шаг:
```
