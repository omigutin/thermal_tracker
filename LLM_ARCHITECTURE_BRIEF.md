# Thermal Tracker — краткий архитектурный гид для LLM

## 1) Архитектурный принцип

Проект построен вокруг потока обработки кадра, а не вокруг UI.

Базовый поток:
`источник кадра -> preprocessing -> tracking/recovery -> результат -> клиент/интеграция`.

Это позволяет запускать одно и то же ядро в разных режимах: desktop и server.

---

## 2) Карта модулей (что где лежит)

## `src/thermal_tracker/core`
Алгоритмическое ядро:
- `config/` — загрузка и модели конфигурации runtime/dev;
- `connections/` — адаптеры источников кадров/команд/результатов;
- `domain/` — доменные модели состояния, снимков и контрактов;
- `processing_stages/` — стадии обработки кадра;
- `scenarios/` — сценарии, которые связывают стадии в рабочий pipeline;
- `storage/` — сохранение истории и связанных артефактов;
- `nnet_interface/` — интерфейсы к нейросетевым моделям.

## `src/thermal_tracker/server`
Серверный слой:
- `cli.py` — CLI вход;
- `runtime_app.py` — основной runtime-цикл обработки кадров;
- `services/` — gateway/runtime-сервисы и инфраструктура запуска.

## `src/thermal_tracker/client`
Клиентский слой:
- desktop GUI;
- web-клиент;
- клиентские сервисы взаимодействия с сервером/runtime.

## Корневые артефакты
- `run_server.py`, `run_desktop_client.py`, `run_web_client.py` — точки запуска;
- `configs/` — конфиги режимов;
- `presets/` — алгоритмические пресеты;
- `trackers/`, `models/` — внешние ресурсы трекинга/моделей;
- `docs/` — основная документация.

---

## 3) Важные runtime-связи

1. Runtime читает конфиг (`configs/server.toml` по умолчанию).
2. Поднимает reader команд, reader кадров и writer результатов.
3. Создает сценарий обработки.
4. На каждом кадре выполняет шаг сценария.
5. Отдает payload результата в транспорт/клиент.

Ключевой файл для понимания серверного цикла:
- `src/thermal_tracker/server/runtime_app.py`.

---

## 4) Как безопасно вносить изменения

1. Не смешивать ответственность слоев:
   - алгоритмы — в `core`;
   - протоколы/запуск — в `server`;
   - UI — в `client`.
2. Новые настройки проводить через dataclass-конфиг + TOML.
3. Изменения стадий делать так, чтобы сценарии оставались совместимыми.
4. Новые диагностические поля добавлять обратно-совместимо для текущих читателей.

---

## 5) Ближайший технический roadmap (LLM-ориентир)

1. **Диагностика стадий**
   - заполнять структурированную диагностику из preprocessing/tracking/recovery;
   - обеспечить читаемый JSONL для разборов инцидентов.

2. **Качество IRST**
   - улучшить критерии выбора кандидата и recovery;
   - снизить ложные потери цели.

3. **Надежность server-режима**
   - укрепить операционные сценарии запуска/остановки;
   - улучшить наблюдаемость latency и состояния pipeline.

## 6) Мини-шаблон prompt для LLM

Используй этот шаблон, когда просишь LLM менять код:

1. Прочитай `LLM_PROJECT_CONTEXT.md` и `LLM_ARCHITECTURE_BRIEF.md`.
2. Уточни слой изменения (`core`, `server` или `client`).
3. Сначала предложи план и риски, потом код.
4. Не меняй публичный API без явного согласования.
5. Для runtime-изменений проверь обратную совместимость payload/конфигов.
