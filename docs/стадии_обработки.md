# Стадии обработки

Обзорный документ по `processing_stages`: какие стадии есть в трекере, в каком порядке они работают, за что отвечает каждая стадия и где искать подробности. Подробные документы лежат в [`docs/processing_stages`](processing_stages/).

## Общая схема

Базовая цепочка для классического OpenCV-сценария:

1. `frame_preprocessing` - привести сырой кадр к рабочему виду.
2. `frame_stabilization` - оценить глобальный сдвиг камеры между кадрами.
3. Дальше сценарий расходится:
   - ручной режим: `target_selection` по клику, затем `target_tracking`;
   - авто-режим по движению: `moving_area_detection`, `target_candidate_extraction`, `candidate_filtering`, затем выбор или сопровождение цели;
   - NN-режим: детекции приходят через интерфейс нейросети, затем используются в сценарии или трекере.
4. `target_recovery` - восстановить цель после потери, если трекер больше не уверен в текущем положении.

Главное правило: стадия не должна чинить чужую задачу. Если выделение при клике захватывает облако, это проблема `target_selection`. Если после потери цель заменяется похожим облаком, это проблема `target_tracking` или `target_recovery`. Если камера резко уехала, сначала нужно смотреть `frame_stabilization`.

## Сценарии и порядок стадий

`opencv_manual`: ручной выбор цели.

Порядок: `frame_preprocessing` -> `frame_stabilization` -> `target_selection` при клике -> `target_tracking`.

Используется, когда оператор сам указывает цель. Это наш основной режим для анализа спорных видео, потому что ошибка выбора цели хорошо видна.

`opencv_auto_motion`: автоматический поиск движущихся областей.

Порядок: `frame_preprocessing` -> `frame_stabilization` -> `moving_area_detection` -> `target_candidate_extraction` -> `candidate_filtering`.

Используется для первичного поиска объектов по движению. Хорош для обзорного режима, но хуже для плотных сцен: несколько близких объектов легко склеиваются в одного кандидата.

`nn_manual`: ручной выбор цели с нейросетевыми детекциями.

Порядок: `frame_preprocessing` -> `frame_stabilization` -> NN-интерфейс -> `target_tracking`.

Используется, когда есть модель и оператор выбирает цель из понятных детекций. Не должен быть обязательным для запуска без модели.

`nn_auto`: автоматическая работа по нейросетевым детекциям.

Порядок: `frame_preprocessing` -> `frame_stabilization` -> NN-интерфейс -> логика выбора/состояния сценария.

Хороший вариант для боевого режима, если модель обучена именно на наших тепловизионных объектах. Без нормальной модели превращается в дорогой источник случайных bbox.

Подробнее: [`scenarios.md`](processing_stages/scenarios.md).

## `frame_preprocessing`

Подробно: [`frame_preprocessing.md`](processing_stages/frame_preprocessing.md).

Задача: подготовить кадр для всех следующих стадий. На вход приходит сырой кадр, на выходе должен быть `ProcessedFrame`: grayscale, BGR-представление при необходимости, служебные версии кадра для детекции и отображения.

Что уже есть:

- `identity` - минимальная обертка без смысловой обработки;
- `thermal` - базовая тепловизионная подготовка;
- `bilateral` - шумоподавление с сохранением границ;
- `clahe_contrast` - усиление локального контраста;
- `percentile_normalize` - нормализация яркости по процентилям;
- заготовки/экспериментальные варианты: AGC-компенсация, gradient-enhanced, temporal denoise.

Какие методы вообще бывают:

- нормализация яркости;
- локальное усиление контраста;
- шумоподавление;
- компенсация авто-подстройки тепловизора;
- временная фильтрация по нескольким кадрам.

Плюсы:

- единое место для борьбы с мутностью, сменой режима тепловизора и шумом;
- следующие стадии получают более стабильную картинку;
- можно сравнивать алгоритмы без переписывания трекера.

Минусы:

- слишком агрессивная нормализация может стереть мелкую цель;
- CLAHE и резкость могут усилить облака, шум и подписи на видео;
- временная фильтрация добавляет задержку и может размазывать быстрые объекты.

Где смотреть при проблемах:

- объект хорошо виден глазами, но алгоритм видит плохо;
- при смене режима тепловизора всё “плывет”;
- мелкая яркая цель пропадает после усиления или сглаживания.

## `frame_stabilization`

Подробно: [`frame_stabilization.md`](processing_stages/frame_stabilization.md).

Задача: оценить движение камеры между кадрами. На выходе получается `GlobalMotion`: сдвиг, доверие к оценке и дополнительные признаки. Эта стадия не должна искать цель; она только говорит, как сдвинулся весь кадр.

Что уже есть:

- `none` / `no_stabilizer` - без стабилизации;
- ECC translation - оценка сдвига;
- ECC affine - оценка сдвига, масштаба и небольшого поворота;
- phase correlation - быстрый глобальный сдвиг;
- feature affine - оценка по ключевым точкам;
- homography - перспективная модель;
- telemetry-assisted - заготовка для внешней телеметрии.

Какие методы вообще бывают:

- phase correlation;
- ECC;
- keypoints/features;
- optical flow по сетке;
- IMU/телеметрия камеры;
- гибрид: телеметрия как прогноз, изображение как уточнение.

Плюсы:

- трекер понимает, что цель не прыгнула, а сдвинулась камера;
- можно ограничивать поиск около ожидаемой позиции;
- особенно важно для Orange Pi/боевого режима, где камера может резко ехать.

Минусы:

- при замутнении камеры стабилизация может дать ложный сдвиг;
- облака, горизонт и подписи могут доминировать над маленькой целью;
- affine/homography тяжелее и иногда хуже простого translation.

Где смотреть при проблемах:

- Калман “улетает”, хотя объект летит ровно;
- после движения камеры ожидаемая зона оказывается не там;
- в логах падает `response` или доверие к сдвигу.

## `moving_area_detection`

Подробно: [`moving_area_detection.md`](processing_stages/moving_area_detection.md).

Задача: найти области кадра, где есть движение или заметное изменение. Это не выбор цели, а грубая маска “здесь что-то происходит”.

Что уже есть:

- frame difference;
- MOG2;
- KNN;
- running average;
- optical flow;
- thermal change detector.

Какие методы вообще бывают:

- вычитание соседних кадров;
- фоновые модели;
- оптический поток;
- порог по тепловому контрасту;
- гибрид движения и яркости.

Плюсы:

- можно автоматически находить движущиеся объекты без клика;
- хорошо работает, когда фон стабилен;
- дешево по вычислениям, если метод простой.

Минусы:

- камера двигается - весь кадр становится “движением”;
- мутность и авто-подстройка тепловизора дают ложные изменения;
- неподвижная или почти неподвижная цель может исчезнуть.

Где смотреть при проблемах:

- авто-режим видит много мусора;
- несколько объектов рядом слипаются уже на маске;
- после сдвига камеры появляется огромная ложная область.

## `target_candidate_extraction`

Подробно: [`target_candidate_extraction.md`](processing_stages/target_candidate_extraction.md).

Задача: превратить маску движения или контраста в отдельные кандидаты на цель. На выходе должны быть bbox/контуры, но еще не подтвержденные цели.

Что уже есть:

- contour extractor;
- connected components;
- blob extractor;
- watershed;
- distance transform.

Какие методы вообще бывают:

- контуры;
- connected components;
- watershed-разделение близких объектов;
- blob detection;
- distance transform для разделения склеенных объектов.

Плюсы:

- отделяет “область интереса” от конкретного объекта;
- позволяет работать с несколькими кандидатами;
- здесь можно решать проблему “три кабана выделились одной рамкой”.

Минусы:

- при плохой маске кандидаты уже приходят испорченными;
- watershed может резать один объект на части;
- connected components легко склеивает близкие яркие объекты.

Где смотреть при проблемах:

- объект выделяется не целиком;
- несколько соседних объектов попадают в один bbox;
- маленькая цель превращается в кусочек объекта.

## `candidate_filtering`

Подробно: [`candidate_filtering.md`](processing_stages/candidate_filtering.md).

Задача: отсеять плохих кандидатов до выбора цели или передачи выше. Это не трекер и не детектор, а цепочка атомарных фильтров.

Что уже есть:

- `area_aspect` - фильтр по площади и пропорциям bbox;
- `border_touch` - отсечение кандидатов у края кадра;
- `contrast` - проверка локального контраста.

Какие методы вообще бывают:

- фильтр по размеру;
- фильтр по форме;
- фильтр по контрасту;
- фильтр по положению;
- фильтр по скорости/направлению;
- фильтр по истории совпадений.

Плюсы:

- простая и понятная цепочка правил;
- удобно включать/выключать фильтры в пресете;
- хорошо подходит для объяснимых ограничений.

Минусы:

- жесткие правила плохо переносятся между “маленькой точкой” и “кабаном”;
- фильтр по размеру легко отбрасывает правильную цель;
- фильтр не должен подменять трекер: он не знает историю цели полноценно.

Где смотреть при проблемах:

- цель найдена, но отбрасывается;
- small target работает на кабанах только частично;
- хочется “универсальный пресет”, но правила слишком узкие.

## `target_selection`

Подробно: [`target_selection.md`](processing_stages/target_selection.md).

Задача: по клику оператора выбрать конкретную цель и построить стартовый bbox. Это критичная стадия для ручного режима: если стартовый bbox плохой, дальше трекер уже сопровождает ошибку.

Что уже есть:

- OpenCV click selector;
- connected component selector;
- threshold region grow;
- watershed seed selector;
- edge-guided selector;
- локальный patch вокруг клика.

Какие методы вообще бывают:

- локальный порог вокруг клика;
- region growing от точки;
- watershed от seed-точки;
- active contour/graph cut;
- edge-guided segmentation;
- маленькая NN-сегментация объекта по клику.

Плюсы:

- оператор дает сильную подсказку: точка цели известна;
- можно отделять цель от соседних объектов лучше, чем в авто-детекторе;
- именно здесь нужно решать “кликнул в кабана, выдели одного кабана, а не стадо”.

Минусы:

- локальный порог может захватить облако или соседний объект;
- слишком маленький patch выделяет только часть цели;
- слишком большой patch цепляет фон и соседние цели.

Где смотреть при проблемах:

- при клике выделяется слишком большая область;
- выделяется только часть объекта;
- нижний объект нельзя нормально выбрать, хотя глазами он виден.

## `target_tracking`

Подробно: [`target_tracking.md`](processing_stages/target_tracking.md).

Задача: сопровождать уже выбранную цель от кадра к кадру. Вход: текущий кадр, прошлое состояние цели и глобальный сдвиг камеры. Выход: `TrackSnapshot` со статусом, bbox, прогнозом и диагностикой.

Что уже есть:

- `opencv_template_point` - гибрид шаблонного поиска, опорных точек и модели движения;
- `nn_yolo` - сопровождение через YOLO-детекции;
- заготовки/альтернативы: KCF, CSRT, MOSSE, MedianFlow, template-only, point-flow.

Какие методы вообще бывают:

- шаблонный поиск;
- correlation filter trackers;
- optical flow по точкам;
- Калман/альфа-бета фильтр;
- particle filter;
- SORT/DeepSORT/ByteTrack/BoT-SORT;
- Siamese trackers;
- гибрид трекера и локального re-detection.

Плюсы:

- учитывает историю движения;
- может переживать короткие пропадания цели;
- может использовать стабилизацию камеры и прогноз.

Минусы:

- если принять облако как измерение, модель движения начнет честно прогнозировать облако;
- шаблон может устареть после замутнения или смены режима;
- Калман не “понимает объект”, он только сглаживает и прогнозирует числа.

Где смотреть при проблемах:

- цель улетела, а трек стоит на облаке;
- после мутности трек перескочил на похожий фон;
- прогноз ускорился, хотя реальный объект летел ровно.

Практический вывод по текущей проблеме: не надо дальше крутить только коэффициенты Калмана. Нужно разделить принятие измерения и прогноз: если измерение далеко от физически правдоподобной траектории и похоже на фон, оно должно уходить в `target_recovery`, а не обновлять скорость трека.

## `target_recovery`

Подробно: [`target_recovery.md`](processing_stages/target_recovery.md).

Задача: искать потерянную цель после того, как обычный трекер потерял уверенность. Это отдельный режим, а не обычное сопровождение.

Что уже есть:

- local template recovery;
- candidate-based recovery;
- global search recovery;
- multi-scale recovery.

Какие методы вообще бывают:

- локальный поиск около прогноза;
- расширяющееся окно поиска;
- re-detection по кандидатам;
- multi-scale template matching;
- поиск по траектории и скорости;
- NN re-identification или class-specific detector.

Плюсы:

- можно явно отделить “сопровождаем” от “ищем заново”;
- можно не портить модель движения плохими измерениями;
- удобное место для стратегии после длительного замутнения.

Минусы:

- широкий поиск легко находит похожие облака;
- глобальный поиск дорогой;
- без хорошего признака объекта recovery превращается в угадайку.

Где смотреть при проблемах:

- цель пропала, потом вместо нее найдено облако;
- после длительной мутности объект снова виден, но система не возвращается к нему;
- зона поиска расширяется, но выбирает не тот объект.

## Где должна решаться проблема “облако вместо цели”

Не в одном месте.

`frame_stabilization`: проверяет, не уехала ли камера. Если сдвиг плохой, трекер получает неверный прогноз.

`target_tracking`: не должен обновлять модель движения сомнительным измерением. Это главный источник разгона Калмана.

`target_recovery`: должен искать цель после потери, но не принимать первый похожий яркий кусок фона.

`target_selection`: должен давать корректный стартовый bbox. Если старт плохой, вся история дальше грязная.

`frame_preprocessing`: должен сделать объект устойчиво видимым, но не усилить облака до уровня цели.

Лучшее направление сейчас: не “еще один коэффициент”, а явная архитектура состояний:

- `TRACKING`: цель уверенно сопровождается, показываем только bbox цели;
- `SEARCHING`: цель потеряна, показываем прогноз и область поиска;
- `RECOVERING`: найден кандидат, но он еще не подтвержден историей;
- `LOST`: цель ушла или данных недостаточно, модель движения не обновляется.

Так мы перестанем кормить Калман случайными облаками. И да, это скучнее, чем “подкрутим на 0.07”, зато не развалится на следующем ролике.
